# 04-rag — RAG 检索增强生成

> 从零实现完整的 RAG Pipeline：文本分块 → 本地 Embedding → 向量存储 → 检索增强生成 → 对话式 RAG

## 学习目标

- 理解 RAG（Retrieval-Augmented Generation）的核心原理和完整流程
- 掌握文本分块（Chunking）策略及其对检索效果的影响
- 使用 @xenova/transformers 在本地生成 Embedding 向量（无需额外 API）
- 使用 ChromaDB 向量数据库进行文档存储和相似度检索
- 实现完整的 RAG Pipeline，对比纯 LLM 与 RAG 增强的效果差异
- 实现对话式 RAG，解决多轮对话中的代词指代问题

## 环境配置

### 1. 安装依赖

```bash
cd 04-rag
npm install
```

`rag-pipeline` 和 `conversational-rag` 依赖 ChromaDB 向量数据库服务（默认 `http://localhost:8000`）。运行前需要先在**单独的终端窗口**启动：

> ⚠️ 注意：npm 的 `chromadb` 包只是 JS 客户端，不含服务端。服务端需要通过 Docker 或 pip 安装。

**方式1：Docker 运行（推荐）**

```bash
docker run -d -p 8000:8000 -v ./chroma-data:/chroma/chroma chromadb/chroma
```

**方式2：pip 安装运行**

```bash
# 建议使用 venv 隔离环境
python3 -m venv .venv
.venv/bin/pip install chromadb

# 启动服务（新终端窗口）
.venv/bin/chroma run --path ./chroma-data
```

首次运行 `embeddings` 或 `rag-pipeline` 时，会自动下载 Embedding 模型（~90MB），通过 hf-mirror.com 镜像下载，模型下载后会自动缓存。

```
### 1. **块大小（Chunk Size）**
- **通用场景**：通常建议 **256~512 tokens**（如 OpenAI 的文本嵌入模型适配此范围）。
- **长文本检索**：若需保留更多上下文，可扩展至 **500~1000 tokens**。
- **代码/结构化文本**：建议更小的块（如 100~300 tokens），以保持逻辑完整性。
- **大语言模型（LLM）上下文窗口**：确保块大小不超过模型最大限制（例如 GPT-4 为 128K，但实际使用中常设为 1K~4K）。

### 2. **重叠大小（Overlap Size）**
- **一般规则**：重叠部分通常为块大小的 **10%~20%**。
- **示例**：若块大小为 500 tokens，重叠可设为 50~100 tokens。
- **关键信息保护**：对于重要内容（如表格、关键段落），可增大重叠以避免信息割裂。

### 3. **调整建议**
- **任务导向**：
  - **问答系统**：较小的块（200~300 tokens）有助于精准检索。
  - **文档摘要**：较大的块（800~1000 tokens）可保留更多上下文。
- **实验验证**：通过测试不同分块策略的检索效果（如召回率、精度）进行优化。
- **工具参考**：
  - **LangChain**：默认块大小 1000 tokens，重叠 200 tokens。
  - **LlamaIndex**：支持动态分块，可根据句子或段落边界调整。

### 4. **注意事项**
- **语义完整性**：避免在句子或段落中间切断文本。
- **模型限制**：嵌入模型（如 BERT）通常有 512 tokens 限制，需相应调整。
- **性能平衡**：块越小检索越精准，但可能丢失上下文；块越大信息越完整，但计算成本更高。
```

## 核心知识点

### 一、为什么需要 RAG？

| 问题 | 说明 | 例子 |
|------|------|------|
| **幻觉（Hallucination）** | 模型会自信地编造不存在的信息 | 问"公司内部报销流程"，LLM 会编一个看似合理但完全虚构的流程 |
| **知识截止（Knowledge Cutoff）** | 训练数据有时间边界 | 无法回答今天的新闻、最新版本文档 |
| **缺乏私有知识** | 无法获取企业/个人专属数据 | 不了解你的代码库、内部文档、业务规则 |

**RAG 的核心思想**：不要让 LLM 凭"记忆"回答，而是先帮它**查资料**，再让它**基于资料**回答。类比人类：开卷考试 vs 闭卷考试。

### 二、RAG 完整架构

```
┌─────────────────────────────────────────────────────┐
│                    离线阶段 (Indexing)                │
│                                                      │
│  文档 ──→ 加载 ──→ 分块(Chunk) ──→ Embedding ──→ 向量DB │
│  PDF/MD/HTML    Chunking      向量化         存储索引  │
└─────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────┐
│                    在线阶段 (Query)                   │
│                                                      │
│  用户问题 ──→ Embedding ──→ 向量检索 ──→ Top-K 结果    │
│                                    ↓                 │
│              LLM ←── Prompt 组装 ←─┘                  │
│               ↓                                      │
│            最终回答（附带来源引用）                      │
└─────────────────────────────────────────────────────┘
```

各环节详解如下：

#### 2.1 文档加载（Document Loading）

不同格式需要不同解析器：
- **Markdown/TXT**：直接读取
- **PDF**：需要 pdf-parse 等工具，注意表格/图片丢失问题
- **HTML**：需要去除标签，保留语义结构
- **代码文件**：保留缩进和结构信息

#### 2.2 文本分块（Chunking）— 最影响效果的环节

分块的质量**直接决定**检索质量。核心矛盾：
- **块太大**：检索精度低（噪声多），浪费 Token
- **块太小**：丢失上下文，语义不完整

三种策略对比（对应 `chunking.ts` 实现）：

```
策略1: 固定大小分块
┌──────────┐┌──────────┐┌──────────┐
│ 300 chars ││ 300 chars ││ 300 chars │  ← 简单粗暴
└──────────┘└──────────┘└──────────┘
         ↕overlap↕
   可能在句子中间截断！

策略2: 递归字符分块（推荐）
先按段落(\n\n)切 → 太大则按句子(。)切 → 还大则按逗号切
┌─────────────┐┌──────────┐┌────────────┐
│ 完整段落1+2   ││ 完整段落3  ││ 完整段落4+5  │  ← 保留语义边界
└─────────────┘└──────────┘└────────────┘

策略3: 语义分块（高级）
用 Embedding 计算相邻句子相似度，在"语义跳变"处切分
[相似度曲线: ████████▁▁████████▁▁▁█████████]
                    ↑ 这里切     ↑ 这里切
```

| 策略 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| 固定大小 | 实现简单，块大小均匀 | 可能截断语义 | 简单场景、快速原型 |
| 递归字符 | 兼顾语义和大小均匀 | 实现较复杂 | **推荐 — 通用场景** |
| 段落分块 | 语义完整性最好 | 块大小不均匀 | 结构清晰的文档 |

**实践建议**：
- 通用场景：**200-500 Token/块**，overlap 10-20%
- 中文内容：用中文标点（。！？）作为分隔符优先级
- Q&A 场景：较小块（200 Token）+ 更多 Top-K
- 摘要场景：较大块（500-1000 Token）

#### 2.3 Embedding（向量嵌入）— 语义理解的核心

Embedding 的本质：将文本映射到高维向量空间，**语义相近的文本距离更近**。

```
"什么是机器学习"  →  [0.12, -0.34, 0.56, ..., 0.78]  (512维)
"ML 的定义"      →  [0.11, -0.33, 0.55, ..., 0.77]  ← 很近！
"今天天气不错"    →  [0.89, 0.23, -0.45, ..., -0.12]  ← 很远！
```

**模型选型对比**：

| 模型 | 维度 | 运行方式 | 中文效果 | 适合场景 |
|------|------|---------|---------|---------|
| `all-MiniLM-L6-v2` | 384 | 本地 | 差 | 英文场景、快速原型 |
| `bge-small-zh-v1.5` | 512 | 本地（**本模块默认**） | **优秀** | 中文场景推荐 |
| `bge-base-zh-v1.5` | 768 | 本地 | **优秀** | 中文生产环境 |
| `text-embedding-3-small` | 1536 | OpenAI API | 好 | 生产（英文为主） |
| `text-embedding-3-large` | 3072 | OpenAI API | 好 | 高精度需求 |
| `BGE-M3` | 1024 | 本地/API | **优秀** | 多语言通吃，综合最强 |

> 💡 本模块 Demo 同时加载英文和中文两个模型进行 PK 对比，直观展示模型选型对中文检索效果的影响。

#### 2.4 向量数据库 — 高效存储与检索

为什么不用普通数据库？因为需要**近似最近邻搜索（ANN）**，传统 B-tree 索引无法高效处理高维向量。

**相似度计算方法**：

```
余弦相似度 (Cosine Similarity) — 最常用
        A · B
cos θ = ─────────    值域 [-1, 1]，1 = 完全相同方向
       |A| × |B|

欧氏距离 (L2 Distance) — ChromaDB 默认
d = √(Σ(ai - bi)²)    值越小越相似

内积 (Dot Product)
dot = Σ(ai × bi)       归一化后等价于余弦相似度
```

**余弦相似度直觉理解**：

想象两支箭从原点射出，余弦相似度衡量的就是夹角大小，**只看方向、不看长度**：

```
        ↗ A "什么是LLM"
       / 30°  ← 夹角小 = 方向接近 = 语义相似
      /
    ──────→ B "大语言模型的定义"     cos 30° ≈ 0.87 ✅

        ↗ A "什么是LLM"
       /
      / 90°  ← 夹角大 = 方向不同 = 语义无关
    |
    ↓ C "今天天气不错"               cos 90° = 0    ❌
```

**计算示例**（假设只有 4 维，实际 512 维原理一样）：

```
A = "什么是LLM"   → [0.8,  0.6,  0.1,  0.0]
B = "大语言模型"   → [0.7,  0.5,  0.2,  0.1]
C = "今天天气好"   → [0.1,  0.0,  0.9,  0.7]

cosine(A, B):
  点积 = 0.8×0.7 + 0.6×0.5 + 0.1×0.2 + 0×0.1 = 0.88
  |A|  = √(0.64+0.36+0.01+0)    ≈ 1.005
  |B|  = √(0.49+0.25+0.04+0.01) ≈ 0.889
  结果 = 0.88 / (1.005×0.889)   ≈ 0.985  ← 非常相似 ✅

cosine(A, C):
  点积 = 0.8×0.1 + 0.6×0 + 0.1×0.9 + 0×0.7 = 0.17
  结果 = 0.17 / (1.005×1.145)   ≈ 0.148  ← 不相似 ✅
```

**为什么 RAG 更适合余弦相似度而非欧氏距离？**

| 场景 | 余弦相似度 | 欧氏距离 |
|------|-----------|---------|
| 同义短文 vs 同义长文 `[0.8, 0.6]` vs `[1.6, 1.2]` | 1.0（方向相同）✅ | 1.0（距离远）❌ |
| 核心区别 | 只看方向，不受长度影响 | 看绝对位置 |

> 💡 本模块 `embeddings.ts` 中调用 pipeline 时设置了 `normalize: true`，Embedding 向量已归一化（模长=1）。归一化后余弦相似度退化为点积运算，计算更快。代码中保留了完整余弦公式以便理解。

**数据库选型**：

| 数据库 | 特点 | 适合场景 |
|--------|------|---------|
| **ChromaDB** | 轻量、纯 JS/Python、内存或本地持久化 | 学习、原型、小规模（本模块使用） |
| **Qdrant** | 高性能、丰富过滤、Rust 实现 | 中大规模生产 |

#### 2.5 向量索引原理 — 如何从百万文档中毫秒级检索

暴力检索（逐一比较）在百万级文档下不可用：

```
文档数 N=100万，维度 D=512
每次查询 = 100万 × 512 次乘法 ≈ 5亿次浮点运算 → 太慢！
```

解决方案：**近似最近邻（ANN）** — 用少量精度换几个数量级的速度提升。

**HNSW（Hierarchical Navigable Small World）— ChromaDB/Qdrant 默认**

构建层级图结构，类似跳表：

```
Layer 2:    A ──────────────── D           (稀疏，长距离跳跃)
            │                  │
Layer 1:    A ──── B ──── C ── D ── E      (中等密度)
            │     │     │     │     │
Layer 0:    A ─ B ─ C ─ D ─ E ─ F ─ G ─ H  (最密，所有节点)
```

检索过程（类比找人）：
```
① 从最高层出发 → 大步跳跃到大致区域（省 → 市）
② 下降一层     → 中步跳跃到更精确区域（市 → 区）
③ 到最底层     → 在邻居间精确比较（区 → 门牌号）

时间复杂度: O(log N)  vs  暴力 O(N)
100万文档: ~20次比较  vs  100万次比较
```

**IVF（Inverted File Index）— Milvus/Faiss 常用**

```
预处理：用 K-Means 把所有向量聚成 n 个簇

     簇1          簇2          簇3
    /    \       /    \       /    \
   •  •  •     •  •  •     •  •  •
   •  •        •  •  •     •  •  •  •

检索时：
① 先找 query 最近的 k 个簇（只比较 n 个质心）
② 只在这 k 个簇内暴力搜索

100万文档, 1000个簇, 搜3个簇 → 只比较 3000 个向量
```

**PQ（Product Quantization）— 压缩存储**

```
原始: 512维 × 4字节 = 2KB/向量，100万文档 = 2GB
PQ:   把512维切成64组，每组用1字节编码 = 64字节/向量，100万 = 64MB
代价: 距离计算变成近似值，精度略降
```

**索引算法对比**：

| 算法 | 查询速度 | 内存占用 | 精度 | 适用场景 |
|------|---------|---------|------|---------|
| 暴力 KNN | O(N) 慢 | 原始大小 | 100% | <1万条 |
| **HNSW** | O(log N) 极快 | 较大（存图） | ~95%+ | **通用首选**，ChromaDB/Qdrant 默认 |
| IVF | 快 | 中等 | ~90%+ | 大规模，Milvus/Faiss |
| PQ | 快 | **极小** | ~85%+ | 内存受限 |
| IVF+PQ | 快 | 小 | ~90% | 亿级数据 |

> 💡 核心权衡：**精度 vs 速度 vs 内存**三者不可兼得。HNSW 在精度和速度上最均衡，是千万级以内的首选。

**ChromaDB 中的实际流程**（对应 `vector-store.ts`）：

```
addDocuments(texts)
│
├─ ① embeddingFunction.generate(texts)    → 批量向量化 (bge-small-zh)
├─ ② 构建 HNSW 索引（每个向量 = 图中一个节点）
└─ ③ 持久化到 chroma-data/

search(query, topK=3)
│
├─ ① embeddingFunction.generate([query])  → 查询向量化
├─ ② HNSW 图检索：顶层 → 逐层下降 → 找到最近的3个节点
└─ ③ 返回 Top-3 文档 + cosine distance
```

#### 2.6 检索策略

**基础：相似度检索（Similarity Search）**
```
query → embedding → 在向量DB中找 Top-K 最近的
```

**进阶：MMR（最大边际相关性）**
```
问题：Top-K 结果可能高度重复（都来自同一段的不同分块）
MMR = λ × 相关性 - (1-λ) × 与已选结果的重复度
效果：既相关又多样化
```

**高级：混合检索（Hybrid Search）**
```
关键词检索(BM25) + 语义检索(Embedding) → 加权融合
优势：兼顾精确匹配和语义理解
"Python 3.12 新特性" → BM25 精确匹配 "3.12"，语义理解 "新特性"
```

#### 2.7 Prompt 组装 — 把检索结果喂给 LLM

```
System: 你是一个基于知识库的问答助手。根据以下参考资料回答问题。
        如果资料中没有答案，请明确说明。

        参考资料：
        [1] {检索结果1}
        [2] {检索结果2}
        [3] {检索结果3}

User:   {用户问题}
```

关键技巧：
- 告诉模型"如果不知道就说不知道"，减少幻觉
- 标注来源编号，便于追溯
- 控制 Top-K 数量，避免上下文窗口溢出

### 三、对话式 RAG — 解决多轮对话难题

单轮 RAG 有一个问题：

```
用户: 什么是 RAG？
助手: RAG 是检索增强生成...

用户: 它有什么优势？     ← "它"指什么？向量检索不理解代词！
```

**解决方案：问题改写（Query Rewriting）**

```
对话历史 + 当前问题 → LLM 改写 → 独立问题 → 正常 RAG 流程

"它有什么优势？"       → 改写为 → "RAG 有什么优势？"
"第一个阶段怎么做？"   → 改写为 → "RAG 的离线索引阶段怎么做？"
```

#### 改写流程详解（对应 `conversational-rag.ts` 的 `rewriteQuestion` 方法）

以 3 轮 Demo 对话为例：

```
第1轮: "什么是 RAG？它解决了什么问题？"
第2轮: "它的完整工作流程是怎样的？"        ← "它" = ?
第3轮: "在第一个阶段中，分块策略有哪些选择？"  ← "第一个阶段" = ?
```

**第1轮 — 无需改写**

```
history = []（空）→ 直接跳过改写，返回原问题
然后存入历史:
  history = [
    { user: "什么是 RAG？它解决了什么问题？" },
    { assistant: "RAG 是检索增强生成..." }
  ]
```

**第2轮 — "它" → "RAG"**

```
① 构建历史摘要（只取最近3轮，每条截断100字）：
   用户: 什么是 RAG？它解决了什么问题？
   助手: RAG 是检索增强生成...

② 组装改写 Prompt → 发给 LLM（temperature=0 保证稳定）：
   "你是一个问题改写助手。根据对话历史，将当前问题改写为独立问题..."

③ LLM 推断 "它" = "RAG"，返回：
   "RAG（Retrieval-Augmented Generation）的完整工作流程是怎样的？"

④ 用改写结果检索：
   原始: "它的完整工作流程"      → 😵 向量检索匹配到无关文档
   改写: "RAG的完整工作流程"     → ✅ 精准匹配到 RAG 流程章节
```

**第3轮 — "第一个阶段" → "离线阶段/Indexing"**

```
历史中有: "...工作流程分为两个阶段：离线阶段（Indexing）和在线阶段..."
LLM 推断 "第一个阶段" = 离线阶段
改写: "RAG在检索阶段的分块策略有哪些选择？"
```

#### 整体数据流

```
chat("它有什么优势？")
│
├─ ① rewriteQuestion ──→ LLM ──→ "RAG有什么优势？"
│       ↑ 读取 history
│
├─ ② ragPipeline.query("RAG有什么优势？")
│       ├─ Embedding → 向量检索 → Top-K 文档
│       └─ 上下文 + 问题 → LLM → 回答
│
├─ ③ 更新 history（存原始问题，非改写后的）
│       history.push({ user: "它有什么优势？" })
│       history.push({ assistant: "RAG的优势..." })
│
└─ ④ 返回 { answer, rewrittenQuestion, contexts }
```

#### 关键设计决策

| 决策 | 为什么 |
|------|--------|
| history 为空时跳过改写 | 没有上下文，改写没意义，省一次 LLM 调用 |
| 只取最近 3 轮（6 条消息）| 太远的历史对当前问题没帮助，且节省 Token |
| 每条消息截断 100 字 | 改写只需理解主题，避免 Prompt 过长 |
| temperature = 0 | 改写要稳定确定性，不要创意发挥 |
| 改写失败则用原问题 | 容错降级，宁可检索不精准也不要报错中断 |
| 存原始问题到历史 | 保持对话历史自然，改写只是给检索用的中间产物 |

#### 性能开销：每次问答需要两次 LLM 调用

```
用户问: "它有什么优势？"
│
├─ 第1次 LLM 调用: 问题改写（几十个 Token，快且便宜）
│   输入: 历史 + 当前问题
│   输出: "RAG有什么优势？"
│
├─ 向量检索（不调用 LLM，毫秒级）
│
└─ 第2次 LLM 调用: 生成回答（几百个 Token，主要开销）
    输入: 检索到的文档 + 改写后的问题
    输出: 完整回答
```

业界常见的优化方案：

| 方案 | 做法 | 优劣 |
|------|------|------|
| **当前方案（双次调用）** | 改写用独立 LLM 调用 | 简单清晰，改写质量高 |
| **合并到一次调用** | 把历史直接塞进 RAG 的 system prompt | 省一次调用，但历史太长会挤占检索上下文的 Token |
| **小模型改写** | 改写用轻量模型，生成用强模型 | 改写便宜快速，生成质量不降 |
| **规则预判** | 检测到代词才调用改写 LLM | 无代词时省一次调用，但规则不一定准 |

### 四、RAG 效果优化方向

> 💡 以下优化方向均有可运行的对比实验，执行 `npm run rag-optimize` 查看效果。

RAG 效果不好时，按以下顺序排查（投入产出比从高到低）：

```
           效果不好？按这个顺序排查：
           ┌───────────────────┐
     ┌──→  │ 1. 分块策略调优     │  ← 最常见的问题根源
     │     └───────────────────┘
     │     ┌───────────────────┐
     ├──→  │ 2. Embedding 模型升级│  ← 换更好的模型
     │     └───────────────────┘
     │     ┌───────────────────┐
     ├──→  │ 3. 检索策略优化     │  ← MMR / 混合检索 / Rerank
     │     └───────────────────┘
     │     ┌───────────────────┐
     ├──→  │ 4. Prompt 优化     │  ← 指令调优
     │     └───────────────────┘
     │     ┌───────────────────┐
     └──→  │ 5. 增加 Reranker   │  ← 对检索结果二次排序
           └───────────────────┘
```

#### 4.1 分块策略调优 — 最常见的问题根源

分块直接决定"检索到的内容质量"，是最值得先调的环节。

**常见问题与解决方案**：

```
问题: 检索到的文档和问题相关，但答案被切断在两个块之间
原因: 块大小太小，或者切分位置不对
解决: ① 增大 chunkSize  ② 增大 overlap  ③ 换递归字符分块

问题: 检索到的文档包含答案，但掺杂了太多无关信息
原因: 块大小太大，信噪比低
解决: 减小 chunkSize，让每个块更聚焦

问题: 同一段内容被多个块重复覆盖，Top-K 结果高度雷同
原因: overlap 设置过大
解决: 减小 overlap，或使用 MMR 检索策略去重
```

**调参建议**：

| 场景 | chunkSize | overlap | 说明 |
|------|-----------|---------|------|
| Q&A 问答 | 200-300 | 30-50 | 小块精准检索 |
| 文档摘要 | 500-1000 | 50-100 | 大块保留上下文 |
| 代码检索 | 按函数/类切 | 0 | 语义完整性优先 |
| 中文通用 | 300-500 | 50-80 | 兼顾精度和上下文 |

#### 4.2 Embedding 模型升级

模型选择直接影响"能不能找到语义相关的文档"。

```
同一个查询 "什么是大语言模型？" 的检索效果：

英文模型 (all-MiniLM-L6-v2):
  "LLM是基于Transformer的模型"    0.30  ← 应该最高，但最低
  "今天天气真好"                   0.60  ← 无关内容反而最高
  → 中文场景完全不可用

中文模型 (bge-small-zh-v1.5):
  "LLM是基于Transformer的模型"    0.56  ← 正确最高 ✅
  "今天天气真好"                   0.20  ← 正确最低 ✅
  → 排序合理，检索有效
```

**升级路径**：

```
学习/原型            → all-MiniLM-L6-v2 (384维, 最快)
   ↓ 中文效果差
中文场景             → bge-small-zh-v1.5 (512维, 本模块默认)
   ↓ 需要更高精度
中文生产             → bge-base-zh-v1.5 (768维) 或 bge-large-zh (1024维)
   ↓ 需要多语言
多语言               → bge-m3 (1024维, 100+语言)
   ↓ 不想自己部署
API 服务             → OpenAI text-embedding-3-small/large
```

#### 4.3 检索策略优化

检索到的文档质量可以通过策略层优化，不需要换模型。

**问题：Top-K 结果高度重复**

```
查询: "RAG的优势"
Top-3 结果（普通相似度检索）:
  [1] "RAG能减少幻觉...RAG的优势在于..."     ← chunk 5
  [2] "...RAG的优势在于...知识时效性..."       ← chunk 6 (和chunk5重叠!)
  [3] "...RAG的优势包括可追溯性..."           ← chunk 7 (还是重叠!)
→ 三个结果说的是同一段内容的不同分块，信息量很低
```

**解决方案1：MMR（最大边际相关性）**

```
MMR 公式: score = λ × 相关性 - (1-λ) × 与已选结果的相似度

λ=1.0 → 纯相关性排序（普通检索）
λ=0.5 → 兼顾相关性和多样性（推荐）
λ=0.0 → 纯多样性（极端，一般不用）

效果:
  [1] "RAG能减少幻觉..."            ← 最相关
  [2] "Embedding是将文本转为向量..."  ← 相关但不重复 ✅
  [3] "文本分块是RAG的关键步骤..."    ← 不同角度 ✅
→ 信息量更丰富
```

**解决方案2：混合检索（Hybrid Search）**

```
问题: 用户搜 "Python 3.12 新特性"

纯语义检索:
  ✅ 理解 "新特性" 的语义
  ❌ 可能匹配到 "Python 3.11 新特性"（语义也很近）

纯关键词检索(BM25):
  ✅ 精确匹配 "3.12"
  ❌ 不理解 "新特性" ≈ "新功能" ≈ "更新内容"

混合检索:
  final_score = α × 语义分数 + (1-α) × 关键词分数
  → 既精确匹配版本号，又理解语义 ✅
```

#### 4.4 Prompt 优化

检索到了好的文档，但 LLM 回答质量差？问题在 Prompt。

**常见问题与优化**：

```
问题: LLM 忽略参考资料，用自己的知识回答
优化: 加强指令约束
  × "请根据资料回答"
  ✅ "仅根据以下参考资料回答。如果资料中没有答案，必须回复'根据现有资料未找到相关信息'，不要编造。"

问题: 回答太笼统，没有引用具体内容
优化: 要求引用来源
  ✅ "回答时用 [1][2] 标注信息来源编号"

问题: 参考资料有冲突信息
优化: 告知处理规则
  ✅ "如果参考资料之间有冲突，请指出差异并说明各自来源"

问题: 上下文窗口放不下所有检索结果
优化: 控制 Top-K 和块大小
  - 减小 topK（3→2）
  - 减小 chunkSize
  - 对检索结果做摘要后再注入 Prompt
```

#### 4.5 增加 Reranker — 精度大幅提升

Reranker 是效果上限最高但成本也最高的优化。

**原理：两阶段检索**

```
阶段1: Embedding 粗筛（快但粗糙）
  查询向量 → 从100万文档中找 Top-20 → 毫秒级

阶段2: Cross-Encoder 精排（慢但精确）
  把 (query, doc) 组成 20 个文本对 → 逐对打分 → 取 Top-3
```

**为什么 Cross-Encoder 更精确？**

```
Bi-Encoder（普通 Embedding）:
  query → [向量A]  }
                    } 分别编码后比较距离，交互信息少
  doc   → [向量B]  }

Cross-Encoder（Reranker）:
  [query + doc] → 一起编码 → 直接输出相关性分数
  query 和 doc 的每个 token 都能互相 attention
  → 理解能力强得多，但无法预计算，每次都要重新推理
```

**代价与收益**：

| 指标 | 无 Reranker | 有 Reranker |
|------|------------|------------|
| 检索精度 | ~70% | ~90%+ |
| 查询延迟 | ~50ms | ~200-500ms |
| 计算成本 | 低 | 中（只对 Top-20 精排） |
| 实现复杂度 | 简单 | 需额外模型 |

常用 Reranker 模型：`bge-reranker-base`、`cohere rerank`、`cross-encoder/ms-marco-MiniLM-L-6-v2`

#### 优化总结

| 优化手段 | 投入 | 效果 | 优先级 |
|---------|------|------|--------|
| 分块策略调参 | 改几个数字 | 高 | ⭐⭐⭐⭐⭐ |
| 换中文 Embedding | 改一行代码 | 高 | ⭐⭐⭐⭐⭐ |
| 距离函数选 cosine | 改一行配置 | 中 | ⭐⭐⭐⭐ |
| Prompt 指令优化 | 改 Prompt 文本 | 中 | ⭐⭐⭐⭐ |
| MMR 多样性检索 | 几行代码 | 中 | ⭐⭐⭐ |
| 混合检索 | 引入 BM25 库 | 中高 | ⭐⭐⭐ |
| Reranker 重排序 | 引入额外模型 | 高 | ⭐⭐ |
